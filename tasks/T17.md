# T17: HD Wallet — Database Migration and DerivedAddress Model

**Status**: todo
**Layer**: backend
**Assignee**: developer
**Depends on**: none (all prior tasks are done)

## Context

The HD wallet feature requires two schema changes on top of the existing `wallets` table and a new `derived_addresses` table. This task creates the ORM model for `DerivedAddress`, extends the `Wallet` model with the `wallet_type` and `extended_key_type` columns, and produces the Alembic migration that applies both changes.

This task is the foundation for every other HD wallet task. No other task may touch `backend/models/wallet.py`, `backend/models/derived_address.py`, `backend/models/__init__.py`, or the migration file concurrently.

> "Two new nullable columns are added to the Wallet model... The existing address field stores the raw xpub/ypub/zpub string for HD wallets."
> — TECH_SPEC_HD_WALLETS.md §4.4

> "File: backend/models/derived_address.py (new file)"
> — TECH_SPEC_HD_WALLETS.md §4.5

## Notes

- The `wallet_type` column must have `server_default="individual"` in the migration so that existing rows get classified correctly without a data migration step.
- The `balance_sat` integer column is required on `DerivedAddress` for correct numeric ordering in `DerivedAddressRepository.get_by_wallet()`. The spec notes this decision in §4.6 — do not omit it.
- SQLite supports `ADD COLUMN` with `server_default`. No manual `UPDATE` statement is needed.
- Alembic must be run with the virtual environment active: `source .venv/bin/activate && alembic upgrade head`.
- This is the only task that may create a migration in this cycle. No other task touches Alembic.
