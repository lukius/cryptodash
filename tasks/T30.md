# T30: Fix BUG-04 — XpubClient includes n_tx=0 entries in derived address list, violating FR-H13

**Status**: todo
**Layer**: backend
**Assignee**: developer
**Depends on**: none

## Context

`XpubClient.get_xpub_summary()` in `backend/clients/xpub.py` (lines 63–70) builds the `derived_addresses` list from every entry in the `addresses` array returned by the blockchain.info `multiaddr` endpoint. However, the API can include entries with `n_tx: 0` — most notably the xpub key string itself is sometimes echoed back as an "address" entry with zero transactions when the wallet has no activity. This violates FR-H13, which defines an active derived address as one with at least one confirmed on-chain transaction.

The effect: users see a spurious entry in the derived address list (the xpub string itself, or a zero-transaction address) with a zero balance, which is confusing and incorrect.

> "The system shall define an 'active derived address' as one that has at least one confirmed on-chain transaction (incoming or outgoing). Derived addresses with zero transactions shall not appear in the list."
> — FUNC_SPEC_HD_WALLETS.md FR-H13

The fix is a single-line filter in `get_xpub_summary` before appending to the derived list.

## Files owned

- `backend/clients/xpub.py` (modify — add `n_tx > 0` filter when building `derived_addresses` in `get_xpub_summary`)
- `tests/backend/test_xpub_client.py` (modify — add regression test)

## Subtasks

- [ ] ST1: In `backend/clients/xpub.py`, in the `get_xpub_summary` method, add an `n_tx > 0` guard when appending to the `derived` list:

  ```python
  for addr in data.get("addresses", []):
      if addr.get("n_tx", 0) > 0:          # FR-H13: active addresses only
          derived.append(DerivedAddressData(
              address=addr["address"],
              balance_sat=addr.get("final_balance", 0),
              n_tx=addr.get("n_tx", 0),
          ))
  ```

- [ ] ST2: Add a regression test to `tests/backend/test_xpub_client.py`:
  - `test_get_xpub_summary_filters_zero_tx_addresses`: mock `multiaddr` response includes one address with `n_tx: 3` and one with `n_tx: 0`; assert the returned `XpubSummary.derived_addresses` contains only the `n_tx: 3` entry (length 1, not 2).
- [ ] ST3: Add a second regression test:
  - `test_get_xpub_summary_xpub_echo_filtered`: mock response includes the xpub string itself as an address entry with `n_tx: 0` (the specific blockchain.info quirk that triggered this bug); assert it is not present in `derived_addresses`.

## Notes

- Severity: Medium. The bug produces incorrect data visible to users — a spurious derived address row in the expanded wallet list. No data is lost or corrupted; the fix prevents bad data from entering the system.
- The filter belongs in `XpubClient.get_xpub_summary` (the ingestion point), not in `DerivedAddressRepository` or the router. Filtering at the source keeps downstream code clean and ensures `XpubSummary` itself is always semantically correct.
- `n_tx` in the `addresses` array reflects total transactions for that derived address. An entry with `n_tx: 0` has never had a transaction and should never appear in the active address list per FR-H13.
- The `aggregate balance` (`wallet.final_balance`) is unaffected by this filter — it is read separately from the `wallet` key, not from the `addresses` array.
