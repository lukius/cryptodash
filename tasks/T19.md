## Context

This task implements the `DerivedAddressRepository`, which manages the `derived_addresses` table. The repository provides two operations: atomically replacing all derived addresses for a wallet (delete-then-insert), and retrieving them ordered by balance descending. It enforces the 200-row cap and returns the total pre-cap count so the service layer can surface the "Showing top 200 of N" note.

## Notes

- The `balance_sat` column on `DerivedAddress` is used for ordering to avoid SQLite string-sorting issues with Decimal strings. See TECH_SPEC_HD_WALLETS.md §4.6 note.
- `replace_all` receives a list of dicts with keys `"address"`, `"balance_btc"` (Decimal), and `"balance_sat"` (int). Generate a UUID4 for each row's `id`.
- The delete-then-insert pattern is used (not upsert) because the full derived address set is replaced wholesale on each fetch. This matches the spec's "cache, not history" framing (§4.5 lifecycle note).
