**Status**: todo
**Layer**: backend
**Assignee**: developer

## Context

This task extends `HistoryService` to support HD wallets at the aggregate level. It adds `full_import_hd()` (one-time history import on first add) and `incremental_sync_hd()` (called on subsequent refreshes to fetch only new transactions). Both methods use `XpubClient` and follow the same replay algorithm as individual wallet history — running-balance reconstruction from oldest-to-newest transactions.

Historical data is aggregate-only. No per-derived-address history is stored.

## Notes

- `full_import_hd` reuses the same internal helpers as individual wallet import (`_batch_insert_transactions`, `_store_daily_snapshots`, `_fetch_and_store_historical_prices`). Do not duplicate this logic.
- Unconfirmed transactions (`timestamp=None`) are stored in the DB with `timestamp=datetime.utcnow()` by `XpubClient._parse_txs`, but are skipped during history replay (`if tx.timestamp is None: continue`) — the `source="live"` snapshot is authoritative for current balance.
- `incremental_sync_hd` must deduplicate against already-stored transactions. If a tx_hash is already in the DB for this wallet, skip it (use the existing unique constraint on `transactions` or check before insert).
- ST5 coordination: T21 (RefreshService) calls `incremental_sync_hd` after the balance fetch. T22 implements the method. If T21 is done first, it should leave a `# TODO: call history_service.incremental_sync_hd(wallet)` stub that T22 completes.
