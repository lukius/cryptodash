# T27: Fix BUG-01 — GET /api/wallets returns empty arrays instead of nulls for HD wallets in loading state

**Status**: todo
**Layer**: backend
**Assignee**: developer
**Depends on**: none

## Context

When an HD wallet is in the loading state (`hd_loading=true`), the `list_wallets()` handler currently queries `DerivedAddressRepository` for every HD wallet regardless of its loading state. This results in `derived_addresses: []`, `derived_address_count: 0`, and `derived_address_total: 0` being returned instead of `null` as required by the spec.

The HD wallet loading state response contract (TECH_SPEC_HD_WALLETS.md §5.2) is:

```json
{
  "balance": null,
  "history_status": "importing",
  "hd_loading": true,
  "derived_addresses": null,
  "derived_address_count": null,
  "derived_address_total": null
}
```

The `[]` vs `null` distinction is semantically significant: `null` means "data not yet available" (loading), while `[]` means "fetch completed and no addresses found." The frontend `DerivedAddressList` component uses `addresses === null` to distinguish the loading spinner from the empty-state message.

> "HD wallet loading state (first fetch in progress)"
> — TECH_SPEC_HD_WALLETS.md §5.2

## Files owned

- `backend/routers/wallets.py` (modify — skip DerivedAddressRepository query and return null fields when `hd_loading=true`)
- `tests/backend/test_wallets.py` (modify — add regression test)

- [ ] ST1: In the `list_wallets` handler in `backend/routers/wallets.py`, add a guard: before calling `derived_address_repo.get_by_wallet(wallet.id)`, check if `hd_loading` is `True` for that wallet. If so, set `derived_addresses=None`, `derived_address_count=None`, `derived_address_total=None` and skip the repository call entirely.
