# T21: HD Wallet — RefreshService Extension

**Status**: todo
**Layer**: backend
**Assignee**: developer
**Depends on**: T18, T19, T20

## Context

This task extends `RefreshService` to handle HD wallets in the balance refresh cycle. It adds `refresh_single_hd_wallet()` (used on first add and in subsequent scheduled refreshes) and modifies the internal `_get_balance()` method (or equivalent) to dispatch to the HD path when `wallet.wallet_type == "hd"`.

The refresh stores an aggregate `BalanceSnapshot`, updates the `DerivedAddress` cache, and optionally writes the `hd_address_count:{wallet_id}` config key when the API returns more than 200 addresses.

## Subtasks

- [ ] ST1: Inject `XpubClient` and `DerivedAddressRepository` into `RefreshService.__init__` (add parameters; update `backend/core/dependencies.py` if needed to wire these in)
- [ ] ST2: Implement `refresh_single_hd_wallet(wallet)` per TECH_SPEC_HD_WALLETS.md §4.8.a:
  - Call `normalize_to_xpub(wallet.address)` to get the API-compatible key
  - Call `xpub_client.get_xpub_summary(xpub)` — on exception, log warning and return `None`
  - Store `BalanceSnapshot(source="live", balance=str(summary.balance_btc), ...)`
  - Build `addr_entries` list with `address`, `balance_btc`, `balance_sat` fields
  - Call `derived_address_repo.replace_all(wallet_id, addr_entries, now)`
  - If `total_count > 200`: store `config_repo.set(f"hd_address_count:{wallet.id}", str(total_count))`
  - Commit and return the snapshot
- [ ] ST3: Modify the balance dispatch loop (the method that fans out to individual BTC/KAS fetches) to check `wallet.wallet_type == "hd"` and call the HD path (`XpubClient.get_xpub_summary` + `replace_all`) per §4.8.b
- [ ] ST4: Implement the wallet-existence guard in the result-writing step (per TECH_SPEC_HD_WALLETS.md §10.8): before writing a `BalanceSnapshot` or calling `replace_all`, check `await wallet_repo.get(wallet_id)`; if `None`, return silently (wallet was deleted during an in-flight fetch). This handles the "remove during active refresh" edge case (FUNC_SPEC §5.1.e) without explicit task cancellation.
- [ ] ST5: Implement the partial-success path for derived address fetch failure (TECH_SPEC_HD_WALLETS.md §10.7 / FUNC_SPEC §5.2.d): if `get_xpub_summary` succeeds but `summary.derived_addresses` is empty AND `summary.n_tx > 0` (API returned balance but no address breakdown), store the `BalanceSnapshot` as normal but skip `replace_all` — leave the existing `DerivedAddress` cache in place. Do NOT delete the existing rows. This allows the old cache to remain displayed with the wallet-level warning icon.
- [ ] ST6: Add HD wallet refresh test cases to `tests/backend/test_refresh.py`:
  - `test_refresh_single_hd_wallet_success`: mocked XpubClient returns summary → snapshot stored, derived addresses replaced
  - `test_refresh_single_hd_wallet_api_failure`: XpubClient raises → returns None, no snapshot stored
  - `test_refresh_full_cycle_includes_hd_wallets`: full refresh includes both individual and HD wallets
  - `test_refresh_hd_wallet_over_200_addresses`: total_count > 200 → config key written
  - `test_refresh_hd_wallet_removed_during_fetch`: wallet deleted between fetch and write → result discarded silently, no DB error
  - `test_refresh_hd_wallet_address_breakdown_missing`: API returns balance but empty addresses with n_tx > 0 → snapshot stored, existing DerivedAddress rows untouched
