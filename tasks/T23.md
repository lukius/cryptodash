## Context

This task updates the Pydantic schemas and the wallet router to expose HD wallet data through the existing `GET /api/wallets` endpoint. No new HTTP routes are needed. The `WalletResponse` schema gains HD-specific fields, and the router's `list_wallets` handler is extended to join derived address data into each HD wallet's response.

## Notes

- `address` in `WalletResponse` contains the full raw xpub/ypub/zpub string. Truncation is performed client-side per ┬¦4.10.a ŌĆö do not truncate in the API.
- `hd_loading: bool` must use the exact derivation from TECH_SPEC_HD_WALLETS.md ┬¦10.10: `True` iff `wallet_type == "hd" AND last_updated is None AND history_status == "importing"`. Do NOT use `wallet.balance is None` alone ŌĆö a failed first fetch also has no balance but is no longer importing. The three-condition formula is the only correct one: it is `True` only during the active initial import window.
- `balance_usd` for each derived address is computed as `balance_native (BTC) * current BTC price`. Reuse the price lookup already used for individual wallets in the list response.
- The `DerivedAddressRepository` must be available in the router scope. Use FastAPI's `Depends()` system consistent with how `WalletRepository` and other repos are injected.
