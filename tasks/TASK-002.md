# T02: Database Models and Initial Migration

**Status**: done
**Layer**: backend
**Assignee**: developer
**Depends on**: T01

## Context

All six SQLAlchemy ORM models must be implemented before any repository, service, or router can
be written. The initial Alembic migration also belongs here: it is the only migration in this
cycle and captures the entire schema. All subsequent tasks depend on these models being stable.

> "All entity IDs are UUIDv4 strings. Timestamps are stored as UTC."
> — specs/TECH_SPEC.md, Section 5.1

## Files owned

- `backend/models/user.py` (create)
- `backend/models/session.py` (create)
- `backend/models/wallet.py` (create)
- `backend/models/transaction.py` (create)
- `backend/models/balance_snapshot.py` (create)
- `backend/models/price_snapshot.py` (create)
- `backend/models/configuration.py` (create)
- `backend/models/__init__.py` (modify — re-export all models)
- `backend/migrations/versions/001_initial_schema.py` (create)
- `tests/backend/test_models.py` (create)

## Subtasks

- [ ] ST1: Implement `User` model — `id` (UUID string PK), `username` (unique, max 50), `password_hash`
      (max 256), `created_at` (DateTime UTC).
- [ ] ST2: Implement `Session` model — `id`, `user_id` (FK → users.id), `token` (unique, indexed, max
      64), `created_at`, `expires_at`.
- [ ] ST3: Implement `Wallet` model — `id`, `user_id` (FK), `network` (max 3: "BTC"/"KAS"), `address`
      (max 128), `tag` (max 50), `created_at`; unique constraint on (user_id, network, address);
      cascade relationships to transactions and balance_snapshots.
- [ ] ST4: Implement `Transaction` model — `id`, `wallet_id` (FK cascade delete), `tx_hash` (max 128),
      `amount` (String 40, signed decimal-as-string), `balance_after` (nullable String 40),
      `block_height` (nullable Integer), `timestamp`, `created_at`; unique constraint on
      (wallet_id, tx_hash); composite index on (wallet_id, block_height).
- [ ] ST5: Implement `BalanceSnapshot` model — `id`, `wallet_id` (FK cascade delete), `balance`
      (String 40), `timestamp`, `source` (max 10: "live"/"historical"); composite index on
      (wallet_id, timestamp).
- [ ] ST6: Implement `PriceSnapshot` model — `id`, `coin` (max 3), `price_usd` (String 40),
      `timestamp`; composite index on (coin, timestamp).
- [ ] ST7: Implement `Configuration` model — `key` (String 64, PK), `value` (String 256), `updated_at`.
- [ ] ST8: Update `backend/models/__init__.py` to re-export all seven model classes.
- [ ] ST9: Write and apply the Alembic migration `001_initial_schema.py` that creates all tables.
- [ ] ST10: Write `tests/backend/test_models.py` — verify each model can be instantiated, inserted, and
       queried via the in-memory test engine.

## Acceptance criteria

- [ ] `alembic upgrade head` applies `001_initial_schema` without error.
- [ ] `alembic downgrade base` reverses the migration cleanly.
- [ ] `pytest tests/backend/test_models.py -v` passes (all 7 model creation/query tests).
- [ ] `from backend.models import User, Session, Wallet, Transaction, BalanceSnapshot, PriceSnapshot, Configuration` imports without error.
- [ ] Wallet cascade: deleting a Wallet row also deletes its Transaction and BalanceSnapshot rows (verified by test).
- [ ] `ruff check backend/models/` exits 0.

## Notes

- Store all Decimal values (amount, balance, price_usd) as `String(40)` per spec Section 5.1 note
  on Decimal storage. Conversion to/from `Decimal` happens at the repository layer, not here.
- IDs are UUIDs generated by the application (`str(uuid4())`), not database auto-increment, per
  TECH_NOTES recommendation cited in spec Section 5.1.
- Only one migration may exist concurrently (Alembic versions are sequential). This task owns the
  only migration in this development cycle.
